# Идеи по решению issue #65 — «Voice Not Working»

> Ссылка: https://github.com/hdrover/discord-drover/issues/65
> Автор: @JanekProduction, 29.04.2026
> Суть: с указанного дня у автора и его друзей голосовой чат через discord-drover не работает — пинг 5000 мс, текстовый чат и остальные функции при этом работают. С альтернативой («гирапом») голос работает. Предположение автора: discord-drover **не проксирует голос**, и именно поэтому голос блокируется.

---

## 1. Корневая причина (краткий анализ)

Это **не баг**, а **архитектурное ограничение** проекта в текущем виде:

- В `drover.dpr` через DDetours хукаются TCP-вызовы (`send`, `WSASend`, `recv`) — для них TCP заворачивается в HTTP/SOCKS5-прокси.
- Для UDP (`WSASendTo`) применяется только **локальная модификация первых байт** (`drover-packet.bin` → `0x00` → `0x01` → `Sleep(50)` → оригинальный 74-байтный IP-discovery пакет). Это эвристика для обхода DPI, **никакого туннелирования нет** — UDP уходит в интернет напрямую с реальным IP клиента.
- Голосовой трафик Discord — это **UDP** (медиа-канал, согласовываемый через WebRTC ICE/DTLS поверх UDP). Если провайдер блокирует UDP до Discord-voice-серверов целиком (а не только по DPI-сигнатуре), никакая локальная мутация первого пакета не поможет — пакеты просто не доходят, и клиент показывает ping = 5000 мс (таймаут).
- «Гирапом» (вероятно имеется в виду GoodbyeDPI / Zapret / Hyperion / другой DPI-обходчик или полноценный VPN/TUN), который у автора работает, либо тоже мутирует UDP, но более удачно, либо это полноценный VPN, заворачивающий **весь** трафик включая UDP в туннель.

Значит, варианты решения распадаются на две большие группы:

- **A.** Улучшить UDP-мутацию (остаёмся в текущей архитектуре, без проксирования UDP).
- **B.** Реально заворачивать UDP в туннель (нужно расширять архитектуру).

Ниже — список идей в порядке нарастающей сложности.

---

## 2. Краткосрочные идеи (без переписи архитектуры)

### 2.1. Сделать UDP-мутацию настраиваемой через `drover.ini`

Сейчас последовательность жёстко зашита в `MyWSASendTo` (`drover.dpr:368-389`):
1. Если есть `drover-packet.bin` — отправить его.
2. Отправить байт `0x00`.
3. Отправить байт `0x01`.
4. `Sleep(50)`.
5. Отправить оригинальный пакет.

Идея: вынести параметры в `drover.ini`, например:

```ini
[udp]
; список фейковых пакетов перед оригинальным, hex-строки через запятую
prefix_packets = 00, 01
; задержка перед оригинальным пакетом, мс
prefix_delay_ms = 50
; перестать вмешиваться вообще
disable = false
; фрагментировать оригинальный пакет на N частей
split_first = 2
```

Это даст пользователям возможность подобрать комбинацию под провайдера без пересборки DLL. Сейчас единственный регулируемый параметр — `drover-packet.bin`, и это бинарь без структуры.

**Где править:** `Options.pas` (новые поля в `TDroverOptions`), `drover.dpr` (`MyWSASendTo`).

### 2.2. Разные стратегии мутации, выбираемые в `drover.ini`

Не просто настройки, а именованные стратегии:

```ini
[drover]
udp_strategy = uae_v1     ; или uae_v2, ru_v1, split, fake_quic, off
```

Под каждую стратегию — свой код в `drover.dpr`. Это упростит поддержку, потому что в реальном мире каждый провайдер ловится по-своему, и одно решение никогда не покрывает всех.

### 2.3. Фрагментация (split) первого пакета

DPI часто ловит сигнатуру в первых N байтах одного пакета. Если разрезать 74-байтный IP-discovery на два UDP-пакета (например, 1+73 байт), сигнатура ломается. Это уже применено в Direct-режиме как `0x00`, `0x01`, но не как split самого пакета.

Добавить опцию: «вместо двух фейковых байт — разрезать оригинальный пакет на 2 части и отправить с микро-задержкой».

### 2.4. Отправлять фейковый «не-Discord» трафик первым

DPI обычно идентифицирует протокол по первому видимому пакету в потоке. Если на новом UDP-сокете первым отправить «похоже-на-QUIC» / «похоже-на-WebRTC-STUN» пакет, DPI может классифицировать поток как разрешённый и пропустить дальнейшие.

`drover-packet.bin` это уже умеет, но:
- Сейчас только один статический файл.
- Хорошо бы шипить **набор готовых** `.bin` под популярные провайдеры (UAE Etisalat, UAE du, RU РКН и т.д.) и иметь возможность ротации.

### 2.5. Мутация не только первого пакета

Сейчас мутация делается только при `IsFirstSend`. Некоторые DPI смотрят первые K пакетов или умеют синхронизироваться позже. Расширить, чтобы первые N пакетов тоже модифицировались (по флагу из конфига).

### 2.6. Принудительный TCP-fallback для голоса (если Discord поддерживает)

Discord client поддерживает fallback голоса с UDP на TCP (через TURN/TURNS), если UDP не проходит. Сейчас drover **не препятствует** этому переключению, но и **не помогает**. Идея:

- Добавить опцию `force_voice_tcp = true`, которая делает первый исходящий UDP-пакет 74 байт «гарантированно битым» (например, шлёт пустой пакет вместо реального). Через несколько таймаутов Discord-клиент сам перейдёт на TCP. А TCP уже идёт через прокси и работает.

Качество голоса при этом ниже, задержка выше, но **голос вообще будет**, что для issue #65 — основная просьба автора.

**Риск:** нужно проверить, что Discord действительно переключается, а не висит на 5000 мс пинга бесконечно (что и происходит сейчас).

## 3. Среднесрочные идеи (всё ещё в рамках одной DLL)

### 3.1. UDP-over-TCP через прокси (полноценный туннель)

Если у пользователя указан HTTP/SOCKS5-прокси для TCP, можно завернуть **UDP внутрь TCP**:

- Хук на `WSASendTo` не делает реальный `sendto`, а создаёт отдельный TCP-сокет до прокси и шлёт пакет туда, инкапсулировав его (длина + данные + sockaddr).
- На стороне прокси нужен сервер, который раскрывает капсулу и шлёт пакет настоящим UDP до Discord-сервера, а ответы возвращает обратно по TCP.
- На клиенте хук `WSARecvFrom` достаёт ответы из TCP-канала и отдаёт Discord.

Это работает, **но требует своего серверного компонента** — это уже не «работаем с любым прокси из интернета». Альтернатива: использовать существующий протокол, см. 3.2.

### 3.2. SOCKS5 UDP ASSOCIATE

SOCKS5 (по RFC 1928) умеет проксировать UDP через команду `UDP ASSOCIATE`: клиент открывает TCP-управляющий канал к прокси, прокси сообщает порт, на который клиент шлёт UDP-пакеты с заголовком (адрес назначения + данные), и прокси релеит их.

Идея:
- В режиме SOCKS5 (`drover.ini`: `proxy = socks5://...`) для каждого нового UDP-сокета Discord:
  1. Открыть TCP-control-канал до прокси, сделать `UDP ASSOCIATE`.
  2. Получить от прокси адрес/порт UDP-relay.
  3. Подменять `WSASendTo`/`WSARecvFrom` так, чтобы реальный отправитель — прокси, реальный получатель — прокси-relay, а адресат закодирован в SOCKS5-UDP-заголовке.

**Это правильное решение для issue #65 при наличии SOCKS5-прокси, поддерживающего UDP ASSOCIATE.** Большинство публичных SOCKS5 не поддерживают, но это уже вопрос выбора прокси, а не реализации.

Сложность: реализация SOCKS5 UDP-relay поверх WinSock-хуков, корректная обработка асинхронных `WSARecvFrom`, синхронизация с control-каналом, fallback, если прокси сказал, что UDP не поддерживает.

**Это самая «честная» фича, которую можно положить в проект.**

### 3.3. Локальный микро-прокси (sidecar)

Если не хотим тащить в Delphi-DLL весь сложный код SOCKS5 UDP-relay, можно:

- Поставить рядом с `version.dll` маленький сервис на Go/Rust/C++ (`drover-helper.exe`), который слушает на 127.0.0.1.
- DLL заворачивает UDP-вызовы в локальный UDP-сокет на 127.0.0.1.
- Сайдкар уже умеет говорить SOCKS5 UDP ASSOCIATE / WireGuard / shadowsocks / любое будущее расширение.

Плюсы: язык DLL остаётся Delphi (минимальные правки), вся сложная сетевая логика на современном языке с нормальной асинхронщиной. Минусы: ещё один процесс, ещё один файл, ещё один потенциальный вектор детектирования антивирусами.

## 4. Долгосрочные / альтернативные идеи

### 4.1. WinDivert / TUN-альтернатива в одном пакете

Завести **опциональный** режим с локальным TUN-устройством (через WinDivert или wintun), который заворачивает UDP конкретного процесса в произвольный туннель. Это противоречит изначальной идее проекта («без драйверов»), но как опт-ин — может быть осмысленно.

### 4.2. Документировать ограничение явно и шипить готовые `drover-packet.bin`

Если решено не делать UDP-проксирование вообще, минимум, что стоит сделать:

- Явно прописать в README, что **голосовой UDP-трафик не проксируется** и работает только через мутацию первого пакета.
- Положить в `dist/` или в отдельный `payloads/` репозиторий **набор подобранных `drover-packet.bin`** под известные ситуации (UAE Etisalat, UAE du, и т.д.) с описанием, какой когда выбирать.
- Закрыть issue #65 как «works as designed» с указанием альтернатив (GoodbyeDPI/Zapret для UDP, полноценный VPN, или SOCKS5 с UDP-relay, если будет реализовано).

### 4.3. Логирование/диагностика

Сейчас в DLL нет ни одного `WriteLn`/лог-файла. Для разбора issue типа #65 нужно либо отладку через Wireshark, либо чужие инструменты. Добавить опциональное логирование (`drover.ini` → `log_file = drover.log`) с информацией:

- Сколько UDP-сокетов создано, сколько `WSASendTo` обработано.
- Был ли первый пакет 74 байт.
- Был ли отправлен `drover-packet.bin`.
- Был ли `recv` после SOCKS5-handshake.

Это сильно ускорит триаж пользовательских жалоб.

## 5. Что я бы делал в первую очередь

Если бы это была моя задача — приоритеты:

1. **(Минимум, можно сделать за день)** Расширить README и issue-ответ: чётко объяснить, что UDP не проксируется, и описать ограничение. Это нужно сделать в любом случае.
2. **(Дни, не недели)** Сделать `udp_strategy` в `drover.ini` (см. 2.2) + опубликовать набор `drover-packet.bin` под популярные провайдеры (см. 4.2). Это закрывает 80% жалоб «у меня не работает после X».
3. **(Среднесрок, недели)** Реализовать SOCKS5 UDP ASSOCIATE (см. 3.2). Это правильное архитектурное решение, и оно вписывается в текущий формат `drover.ini` без новых процессов и драйверов.
4. **(Опционально)** Force-TCP-fallback (см. 2.6) как переходное решение — потому что у пользователя ситуация «голос либо не работает, либо плохо» лучше, чем «голос не работает».

Пункты 1 + 2 закрывают issue #65 с точки зрения пользователя (либо через мутацию работает, либо есть документация и понятная альтернатива), пункт 3 убирает root cause.

## 6. Что точно НЕ стоит делать

- **Не нужно** делать ASIO-подобный пере-движок сокетов в DLL. Текущая DDetours-схема — это правильная архитектура для данной задачи, не ломайте её ради «универсальности».
- **Не нужно** тащить в проект ffmpeg/Opus и пытаться декодировать-перекодировать голос. Discord сам разберётся с кодеками — нам нужно только довезти UDP до сервера.
- **Не нужно** обещать «работает у всех» — каждый провайдер ловит по-своему, и проект должен это явно признавать.
